# Coherence

コードを 1 行でも変えるなら、このスキルを通す。指示された箇所「だけ」を反映して周辺との整合を崩すのが、Claude が最も犯しやすいミス。それを機械的に防ぐ。

## いつ使うか

**コードを編集・追加・削除・リネーム・リファクタするすべての作業で、着手前と完了後に通す。** 「言われた通りに直す」だけでは足りない。変更が周辺に与える波及を必ず検査する。

例外: フォーマットやコメントのみの変更は、ステップ 2 で「フォーマット / コメントのみ」と判定した上でスキップしてよい（理由は報告に残す）。

## ステップ 1: スコープ見積もり（変更前）

着手前に「何を grep / 探索すべきか」を網羅的に列挙する。以下のカテゴリを **必ず一通り考える**。該当しないものは「該当なし」と明示してスキップする。沈黙してスキップしない。

- **識別子**: シンボル名（関数 / 変数 / クラス / 型 / 定数）の全文検索。**部分一致** も確認（接頭辞・接尾辞バリアント、リネーム漏れ）。動的参照（文字列リテラル経由、reflection、`eval`、template literal 内）も。
- **型・スキーマ**: 型定義ファイル (`*.d.ts` / `types/` / `*.proto` / `schema.graphql` / `openapi.yaml`)、`import type`、generated コード（migration / proto / OpenAPI）と元定義の一致。
- **import / export**: 全呼び出し元、re-export（barrel file `index.ts`）、default export と named export の混在。
- **設定 / 環境**: `.env` 系（キーのみ）、`config/*`、`*.toml` / `*.yaml` / `*.json`、`tsconfig.json`、bundler 設定、CI、環境変数を消費している箇所。
- **テスト**: `*.test.*` / `*.spec.*`、`__tests__/`、`__snapshots__/`、フィクスチャ、スナップショットの陳腐化。
- **ドキュメント**: README、ADR、コメント、JSDoc / TSDoc、Storybook、サンプル、tutorial、公開 API ドキュメント。
- **i18n**: 翻訳キーの参照箇所、未翻訳・余剰キーの両方。
- **DB / 永続データ**: migration、seed、スキーマ宣言、既存データとの整合。
- **API 契約**: クライアント側の型・URL・パラメータ、サーバー側のルート定義・バリデーション、共有 SDK や OpenAPI 生成物。
- **依存方向**: 変更対象に依存するモジュール（呼び出し元）、変更対象が依存するモジュール（呼び出し先）。

成果物: 「**探索した識別子・カテゴリの一覧**」と「**理由付きでスキップした領域**」。

## ステップ 2: 影響半径の判定（変更種別マトリクス）

「全部見る」も「最小限」も避け、変更の **意味的種別** から駆動して調査の深さを決める。

| 変更種別 | 必須調査範囲 |
|---|---|
| **リネーム** | 識別子の部分一致全文検索、動的参照、ドキュメント、test 名、コミットメッセージ慣習。**旧名が文字列で残っていないか必ず再 grep** |
| **シグネチャ変更**（引数増減・型・戻り値） | 全呼び出し元、型定義、mock、test、ドキュメント、API 契約。引数追加でデフォルト値があっても `undefined` 起因の挙動変化を確認 |
| **挙動変更**（同シグネチャ・内部仕様変更） | テスト、呼び出し元の前提（コメント / assert）、関連ドキュメント、エッジケース、隠れた呼び出し元の文書検索 |
| **削除** | 全参照、設定キー、docs、test、generated コード、deprecated 警告と移行ガイド。**深いなら `ideal` skill にバトン** |
| **追加** | 既存の類似機能との重複、命名規約、export / barrel の更新、test の追加 |
| **設定キー変更** | 全環境ファイル、CI、デプロイ環境、ドキュメント、README。環境ごとに値が違う場合は **値の整合** も |
| **依存ライブラリ変更** | `analyze-library-update` skill に委譲（重複実装しない） |
| **フォーマット / コメントのみ** | スキップ可。**スキップ理由を報告に明記** |

ステップ 1 で出した「探索対象一覧」を、このマトリクスで **絞り込み** または **拡張** する。

## ステップ 3: 整合性検査（変更後）

修正完了直後に必ず実行する。

1. **再 grep**: 旧名・旧シグネチャ・旧設定キーが残骸として残っていないか、ステップ 1 と同じ検索を再実行。
2. **型チェック**: プロジェクトの型チェッカ（`tsc --noEmit` / `mypy` / `pyright` / `go vet` 等）を完走させる。通せない場合は理由を報告。
3. **lint / formatter**: プロジェクトの lint と formatter を流す。
4. **test**: 影響範囲のテストを実行。範囲が判断できないなら全 test。失敗・skip があれば報告。
5. **docs 整合**: ステップ 2 で対象になった docs / README / コメントが実際に反映されているか目視確認。
6. **生成物の再生成**: codegen / migration / OpenAPI / 翻訳辞書など、生成系の更新有無を確認。
7. **動的参照の最終確認**: 文字列リテラル経由の参照を `rg -F "<旧名>"` で再走査。

**どれもスキップしてよい場合があるが、スキップは必ず理由付きで報告に書く。** 沈黙してスキップしない。

## ステップ 4: 報告

ユーザーに「**どこを見て、何を直し、何を直さなかったか（理由付き）**」を簡潔に伝える。冗長にしない。

```
## 整合性チェック結果

### 変更種別
<リネーム / シグネチャ / 挙動 / 削除 / 追加 / 設定 / その他>

### 調査スコープ
- 探索した識別子: `XXX`, `YYY`
- 検査したカテゴリ: 識別子 / 型 / import / 設定 / test / docs / i18n / 生成物 / API
- 明示的にスキップしたカテゴリ:
  - i18n: 当該機能は未翻訳のため対象外

### 修正一覧
- `path/a.ts:12-30` — 呼び出し側のシグネチャ更新
- `docs/api.md:45` — パラメータ表

### 直さなかった箇所と理由
- `legacy/old-impl.ts:88` — deprecated 領域、別 PR で対応

### 検査結果
- type check: pass
- lint: pass
- test: 24 pass / 0 fail
- 残骸 grep: なし
```

## よくある罠

- **指示された箇所「だけ」直す**: 言われた一行だけ書き換えて呼び出し元を放置する。指示を狭く読みすぎ。
- **テスト無視**: 「コードは直したから OK」で test を回さない。test が古い仕様を要求していたことに気づかない。
- **隠れた依存の見落とし**: 文字列リテラル / 動的 require / template 内の埋め込みを見ない。
- **生成物の同期忘れ**: スキーマだけ直して codegen を走らせない。
- **「直したつもり」報告**: 「全部対応しました」と言いつつ、実際は対象を絞っていた。**スキップは必ず明示**。
- **過剰調査で着手が遅れる**: 種別が「フォーマットのみ」でも全スコープを舐める。**変更種別マトリクスで止める**。

## 他 skill との連携

- **`ideal`** — 削除や大規模シグネチャ変更で「そもそも何が正しいか」を再考すべきと判明したら委譲。
- **`rethink`** — 報告後に「これでよかったか」を見直すフェーズで使う。
- **`best-practice`** — ライブラリ仕様への依存が広い変更で根拠取り。
- **`analyze-library-update`** — 依存ライブラリ変更は本 skill では深追いせず委譲。
