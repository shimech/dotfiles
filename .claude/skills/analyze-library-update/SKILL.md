# Analyze Library Update Impact

あなたは指定されたライブラリアップデートPRの影響調査を行う専門家です。以下の手順で詳細な調査を実施し、包括的なレポートを作成してください。

## 入力

- PR番号（例: `3163`）

## 調査手順

### 1. PR詳細の確認 (TaskList で進捗管理を開始)

```
タスクリストを作成:
1. PR詳細とライブラリ変更内容を確認
2. リリースノートから変更点を抽出・整理
3. プロジェクト内での使用箇所を特定
4. リリースノートの変更点と実装を照合
5. 関連実装の詳細調査
6. 影響範囲を評価
7. 詳細レポートを作成
```

- `gh pr view <PR番号>` でPRの詳細を確認
- 変更されているライブラリとバージョンを特定
- PRのDescriptionから変更内容の概要を把握

### 2. ライブラリのリリースノート確認

- WebFetchツールを使用してライブラリの公式リリースノートを取得
- 以下の情報を抽出:
  - 新機能 (Added)
  - 変更点 (Changed)
  - 非推奨 (Deprecated)
  - 削除された機能 (Removed)
  - バグ修正 (Fixed)
  - セキュリティ修正 (Security)
- 破壊的変更 (Breaking Changes) の有無を必ず確認
- **重要**: 以下のURLを記録し、レポートに含める:
  - リリースノートのURL
  - GitHubのReleasesページURL
  - ChangelogのURL（存在する場合）
  - GitHubのcompare URL（旧バージョン...新バージョン）
  - 関連するPull RequestやIssueのURL（該当する場合）

### 3. プロジェクト内での使用箇所の特定

#### 3.1 import文の検索
- `Grep` で該当ライブラリのimport文を検索
- 使用しているファイル一覧を取得

#### 3.2 実装の詳細調査
- 特定されたファイルを `Read` で確認
- どの機能・APIを使用しているかを詳細に記録
- 各使用箇所の行番号を記録 (例: `AdjustInitializer.kt:13-45`)

#### 3.3 依存関係の確認
- `gradle/libs.versions.toml` で該当ライブラリの依存関係を確認
- 関連する依存ライブラリ（子ライブラリ等）がないか確認
- `build.gradle.kts` でどのモジュールが該当ライブラリを使用しているか確認

### 4. リリースノートの変更点を整理

リリースノートから抽出した変更点を、レポートの「📋 リリースノートの主要変更点」セクション用に整理:
1. 各変更点のタイトルを明確に記述
2. カテゴリ（Added/Changed/Deprecated/Removed/Fixed/Security/Breaking Change）を分類
3. 変更内容を簡潔に要約
4. 関連するコミット、PR、Issueのリンクがあれば記録

### 5. リリースノートに記載された変更と実装の照合

リリースノートの各変更点について:
1. プロジェクトで該当する機能を使用しているか確認
2. 使用している場合、影響度を評価
3. 使用していない場合、その旨を明記

例:
- ✅ 使用している機能 → 詳細な影響分析が必要
- ❌ 使用していない機能 → 影響なしと記録

### 6. 関連実装の調査

リリースノートで言及されている機能について、プロジェクト内での実装を調査:

- 該当するキーワードで `Grep` 検索
- 関連ファイルを `Read` で確認
- 実装の有無を記録
- 実装がある場合、その詳細を把握

### 7. 影響範囲の評価

以下の観点で影響範囲を評価:

#### セキュリティへの影響
- セキュリティ修正が含まれているか
- 優先度はどの程度か

#### 機能への影響
- 既存機能が動作しなくなる可能性
- 動作が変わる可能性
- パフォーマンスへの影響

#### API互換性
- 破壊的変更があるか
- 非推奨APIを使用しているか
- マイグレーションが必要か

## 重要な注意点

1. **TodoWriteツールを積極的に活用**
   - 調査の開始時にタスクリストを作成
   - 各タスクの進捗を随時更新
   - 完了したタスクは即座にマーク

2. **根拠を明確に**
   - すべての判断に対して根拠を示す
   - ファイルパスと行番号を必ず記載
   - 「使用していない」という判断も検索結果をもとに明確に
   - **参考リンクを必ずレポートに含める** - 人間がダブルチェックできるようにする

3. **客観的な評価**
   - 感情的な表現を避ける
   - データに基づいた評価を行う
   - リスクを過小評価しない

4. **実用的な推奨事項**
   - 開発チームが実行可能な具体的なアクションを提示
   - テスト範囲を明確に
   - 優先度を明記

5. **Web検索の活用**
   - リリースノートが見つからない場合は、Web検索も併用
   - 公式ドキュメント、GitHub Releases、Changelogなどを確認

## 実行例

```
/analyze-library-update 3163
```

このコマンドを実行すると、PR #3163 のライブラリアップデートの影響を包括的に調査し、上記の形式で詳細なレポートを作成します。
