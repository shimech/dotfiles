# PR Monitor

PR を「その時点のレビューコメント対応済み」かつ「CI グリーン」まで持っていくスキル。

レビューコメントをずっと polling し続けるのは負荷が高いので、コメント対応はワンパス。継続監視のループは CI 側で回す。

## ワークフロー

1. **pr-feedback をワンパス実行（初回）**
   Skillツールで `pr-feedback` を呼び出し、現時点の未対応コメントに対応する。
   - 対応するコードがある場合はコミットまで行う（push はまだしなくてよい。次のCI確認時にまとめて push でも可）

2. **ci-monitor で CI グリーンまで監視**
   Skillツールで `ci-monitor` を呼び出し、最新 CI が success になるまで面倒を見る。
   - 失敗 → 修正 → push → 再監視 のループは ci-monitor に任せる
   - 1 のコメント対応によるコミットがあれば、ここで push されて新しい CI run が走る

3. **pr-feedback をワンパス実行（最終確認）**
   CI green 到達時点でもう一度 `pr-feedback` を呼び、CI 完走中に届いたコメントを拾う。
   - 新しい対応コミットが発生したら 2 に戻る（新しい CI run を監視）
   - 新しい未対応コメントがなければ完了

この「CI 完走 → コメント最終確認 → 対応があれば CI 再監視」のループでクローズする。ただし CI 完走中ずっと pr-feedback を polling することはしない（負荷が高いので）。

## 完了条件

- 最新の CI が success
- かつ最終確認時点で未対応コメントなし

あとからさらに新しいコメントが届いても、このスキルでは追従しない（追従したい場合はユーザーが再実行）。

## 各サブスキルとの役割分担

### pr-feedback に任せること（ワンパス）
- レビューコメント（インライン／通常）の取得
- 未対応コメントの整理・対応案提示
- 修正のコミット、bot への返信

人間コメントへの返信は pr-feedback のルールに従い **勝手に返信しない**。

### ci-monitor に任せること（継続）
- CI ステータスの取得と判定
- 失敗原因の分析・修正・push
- 実行中のポーリング
- success まで継続監視

## 引数の扱い

`$ARGUMENTS` は各サブスキルが解釈する。pr-monitor 自体では解釈せず、そのまま pr-feedback / ci-monitor に渡す（例: `--branch`, `--repo` 指定）。

## 注意事項

- 修正コミットは原則「1修正1コミット」（各サブスキルの方針に従う）
- 同じ CI が 3 回以上失敗、または同じコメントで判断に迷う場合は、ユーザーに状況を説明して指示を仰ぐ
- ワンパス対応中にユーザーが「コメント再チェック」を求めた場合は、再度 pr-feedback を呼び出す
