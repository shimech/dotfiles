# Best Practices

このスキルの核心は **最新のソースに当たって、今の正解を確認する** こと。内蔵知識を信用せず、Webで裏付けを取る。

## 調査の進め方

### 1. 公式は深く、事例は広く浅く

**必ずWebSearchする。** 内蔵知識だけで回答しない。

**公式ソース（深く読む）:**
- 公式ドキュメントのベストプラクティスページ、ガイド、FAQをWebFetchで本文まで読む
- **CHANGELOG/リリースノートは必ず確認する** — breaking changeや非推奨化の情報がここにある
- 公式ブログやRFCがあれば、推奨が変わった経緯を追う
- **ソースコードやリポジトリも見る** — 公式リポジトリのexample/、test/、README、設定ファイルのデフォルト値など。ドキュメントに書いてない実装上の制約や意図がわかることがある

**事例ソース（広く浅く）:**
- 英語と日本語の両方で検索する
- `"best practices"` だけでなく `"pitfalls"`, `"anti-pattern"`, `"migration guide"` でも探す
- 検索結果のタイトルとスニペットで関連性を判断し、有望なものだけWebFetchする。全部読む必要はない
- 同じことを言っている記事を3つ見つけたらそのトピックは十分。次に進む

### 2. 鮮度を技術の変化速度で判断する

| 変化速度 | 例 | 許容できるソースの古さ |
|----------|-----|----------------------|
| **超高速** | AIツール・LLMフレームワーク、新興SaaS | 1-3ヶ月 |
| **高速** | フロントエンドフレームワーク、クラウドサービス | 6ヶ月-1年 |
| **中速** | 言語のエコシステム、DB運用 | 1-2年 |
| **枯れた技術** | POSIX、SQL標準、TCP/IP | 年数は問わない |

十分な数のソースが集まったら、最新を重視する。

### 3. 複数のアプローチがある場合

有力な選択肢が複数あれば併記し、優先順位を付ける。「これが唯一の正解」と断言できないなら、ユーザーに選択肢を見せる。

優先順位の付け方:
- 公式が推奨しているもの > コミュニティで広く使われているもの > 新しいが実績が少ないもの
- 現プロジェクトの既存構成との親和性も考慮する

判断が分かれるものは、その場で試せるなら試す（設定変更して動作確認など）。試せないなら選択肢とトレードオフを示してユーザーに確認を取る。

## 出力に含めるべきもの

フォーマットはトピックに合わせて自然に選ぶ。ただし以下は必ず含める:

- **結論を最初に**
- **各プラクティスの根拠とソース** — リンク+公開日。リンクなしの「一般的に〜」は禁止
- **やってはいけないこと**
- **現プロジェクトへの適用** — 今のコードとの照合結果。ユーザーが古い方法を使っていたら、何が変わったかもここで伝える

加えて、トピックに応じて:

- 設定系ならBefore/Afterのコード例やコピペ可能な完成形
- ツール系なら検証方法（改善後にどう確認するか）
- 選択肢があるなら比較表
