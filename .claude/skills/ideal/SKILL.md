# Ideal

このスキルが呼ばれたら、手を止めて「そもそも何が正しいのか」を考える。

## ステップ

1. **問題を定義する** — 今解決しようとしているのは何か。表面の症状ではなく、根にある原因は何か。「何を変えるか」ではなく「なぜ困っているのか」から始める

2. **概念を整理する** — 登場する概念の意味・責務・関係を明確にする。名前と実態がズレていたら、それ自体が構造の歪み。一つのものが複数の関心事を持っていないか、一つの関心事が散らばっていないか

3. **理想形を描く** — 制約を一旦忘れて「ゼロから作るなら」を考える。理想の境界線、理想の役割分担、理想のデータの流れ。移行コストや後方互換性はこの段階では考えない

4. **ギャップを明示する** — 理想と現状の差を具体的に示す。そのうえで「理想に近づける現実的なステップ」を提案する。理想を知ったうえで妥協するのと、知らずに場当たり的に対処するのは全く違う

## よくある罠

以下に気づいたら、このスキルの出番:

- **症状を塞いでいる** — エラーが出る箇所にガード条件を足す、フラグを追加する、特殊分岐を増やす。問題が起きている理由ではなく、問題の見え方だけを消そうとしている
- **作業量で判断している** — 「こっちのほうが変更が少ないから」で案を選んでいる。変更量の少なさは設計の正しさと無関係
- **既存構造を正当化している** — 「今こうなっているから」を理由に提案している。既存コードは参考にはするが、正当化の根拠にはならない
- **名前を疑っていない** — 変数名・関数名・ファイル名を所与として受け入れ、そのまま積み上げている。名前が間違っているなら構造も間違っている可能性が高い
